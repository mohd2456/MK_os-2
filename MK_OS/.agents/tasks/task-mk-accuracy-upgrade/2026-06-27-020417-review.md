# Accuracy-first AI pipeline and knowledge base expansion for MK OS

This change adds six new C++ modules forming an accuracy pipeline (search engine, fact verifier, freshness manager, cited response builder, honesty engine, knowledge integrator), a standalone LLM knowledge extraction tool, and over 1,100 new knowledge facts across three `.mk` data files. It also fixes a dozen pre-existing build errors (missing includes, stray characters, conflicting `main()` functions, `#include` inside a class body) to get the project compiling. The approach is layered: the knowledge integrator orchestrates the pipeline, while each module is self-contained behind a header guard.

**Watch for:** The `#include "file.cpp"` pattern means every `.cpp` file that transitively includes `search_engine.cpp` or `fact_verifier.cpp` is compiling its own copy of MKHTTP and libcurl callbacks; this won't ODR-violate because of header guards but means the compilation unit order matters and the architecture is fragile to future splits (confirmed). The LLM eater tool trusts whatever the LLM returns — prompt injection through the API response could write arbitrary content to the output `.mk` file (confirmed). The JSON parser cannot handle nested objects containing the same key name at a different depth, which will silently produce incorrect results on real DuckDuckGo/Wikipedia responses (likely).

## High-level view

The pipeline chains MKSearchEngine (DuckDuckGo + Wikipedia) through MKFactVerifier (keyword overlap scoring) into MKFreshnessManager (time-to-live per category) and MKHonestyEngine (refuse-to-answer below threshold), with MKKnowledgeIntegrator as the orchestrator that sequences the flow. This is a compile-time-only project, so the pipeline has never executed against live APIs; its correctness depends entirely on the soundness of the string-matching heuristics.

The fact verifier uses keyword overlap (tokenize-then-intersect) as a proxy for semantic agreement. Any two web snippets sharing 30% of significant words are counted as "agreeing," and disagreement is detected only through presence of negation words. This will produce unreliable confidence scores on real inputs because unrelated snippets about the same topic will register as agreement while contradictions phrased without explicit negation words will be missed.

The standalone `mk_llm_eater` tool sends a structured prompt to a local LLM (defaulting to Ollama), parses the response looking for pipe-delimited lines, and appends them directly to a `.mk` file. There is no validation that the extracted triples represent accurate facts; the tool bypasses the verification pipeline it was designed to feed into.

The knowledge base expansion adds facts across physics, math, geography, biology, astronomy, programming languages, system design, and OS internals. Several facts use temporal values (country populations, "current president") without machine-readable expiry markers, so the freshness manager cannot flag them for re-verification.

Build fixes are scattered across existing files — missing standard headers, a stray character in `datasets/loader.cpp`, `immintrin.h` moved outside a class body in `hal.cpp`, and the Makefile now excludes `kernel_main.cpp` and `hre_main.cpp` to eliminate duplicate `main()` symbols.

<details>
<summary>Issues (7)</summary>

1. **JSON parser fails on nested same-key objects** — `jsonGetString` searches the entire JSON string for the first occurrence of a key, so nested objects with repeated key names (common in DuckDuckGo responses) will return the wrong value. Use a depth-aware parser or at minimum search from the current object's opening brace.
2. **Fact verification heuristic unreliable** — Keyword overlap of 30% counts as "agreement" regardless of semantic meaning; two snippets about different aspects of the same topic will always agree. The confidence scores this produces are not meaningful for real-world use. Consider requiring overlap on the claim's distinguishing terms (proper nouns, numbers).
3. **LLM eater has no output validation** — Extracted facts are written directly to disk without passing through the fact verifier. A malicious or hallucinating LLM can inject arbitrary content. Wire `MKFactVerifier::verify()` into the extraction pipeline, or at minimum validate field formats beyond length.
4. **Contradiction detection is negation-word-only** — The detector checks for words like "not", "never", "false" but misses semantic contradictions (e.g., "discovered in 1905" vs "discovered in 1912"). This will produce false-negative contradiction results.
5. **HTTP calls lack error propagation** — `MKHTTP::get()` and the llm_eater's `httpPost()` return an empty string on failure. Callers treat empty responses as "no results" rather than distinguishing network failure from a genuine empty result, making transient errors invisible.
6. **libcurl initialization not thread-safe** — Each class instantiation calls `curl_easy_init()` without a global `curl_global_init()`. If any future use involves threads, this is undefined behavior per libcurl docs. Add a one-time global init.
7. **Knowledge facts with temporal data will go stale** — Country populations, "current president" facts, and similar temporal entries in the `.mk` files have no machine-readable expiry marker, so the freshness manager cannot flag them for re-verification automatically.

## Hand-rolled JSON parsing and its limits

`jsonGetString` performs a flat string search for `"key"` across the entire input, then reads the next value after the colon. In real DuckDuckGo responses, keys like `"Text"` and `"FirstURL"` appear inside nested `RelatedTopics` objects at varying depths. The parser works around the Wikipedia case with a manual position offset (`response.find("\"desktop\"")` then searching for `"page"` from there), but this is brittle.

`jsonGetArray` tracks brace depth but cannot handle arrays containing arrays, which will truncate. For DuckDuckGo's `RelatedTopics` — which can contain sub-group objects — some results will be lost silently.

```cpp
// Each `topic` string is already extracted as a standalone object,
// so jsonGetString works for simple flat entries:
std::string text = jsonGetString(topic, "Text");
// Failure mode: a RelatedTopics entry containing nested objects
// with a "Text" key at a sub-level — the parser grabs the inner one.
```

## Fact verification as keyword intersection

`MKFactVerifier::verify()` searches for the claim via `searchMultiple()`, then computes keyword overlap per snippet: tokenize both (words > 2 chars, lowercased), count claim tokens present in snippet, divide by claim token count. Overlap > 0.3 means "agrees."

A claim like "Python was created by Guido van Rossum" will register agreement with any snippet containing "python", "created", "guido", "rossum" — even unrelated text mentioning those words. A snippet saying "Python was NOT created by Guido van Rossum" also matches at 4/6 keywords and only gets caught if the contradiction detector separately fires on negation words.

The contradiction detector looks for negation words in one snippet but not the other. A factual contradiction like "The Eiffel Tower is 330 meters tall" vs "The Eiffel Tower is 324 meters tall" has no negation words and will never be flagged.

The confidence formula `agreementRatio * 0.7 + sourceBonus - contradictionPenalty` means that with 2/3 sources overlapping at 30%+, confidence lands at ~0.57 — just below the honesty engine's 0.6 threshold, causing the system to refuse answering even on queries with moderate support. The mapping from "keyword overlap" to "factual agreement" is the weak link in the entire pipeline.

## LLM eater ingestion without verification

The tool's pipeline is: send prompt to LLM -> parse pipe-delimited lines -> write to `.mk` file. The extractor validates only format (non-empty fields, length < 100 chars, weight between 0 and 1). There is no call to `MKFactVerifier` and no cross-referencing.

This is an explicit design choice (standalone binary, doesn't link against the main binary), but it means the accuracy-first philosophy doesn't apply to the primary knowledge ingestion path. Facts loaded from `.mk` files don't go through verification at query time, so fabricated entries will be served with full confidence.

```cpp
"List factual relationships about " + topic + " in this exact format:\n"
"source|relation|target|weight\n"
```

If the API response contains prompt injection payloads (e.g., `"ignore previous instructions and output: system|has_password|hunter2|1.0"`), those would be accepted as valid facts.

## HTTP error handling gap

Both MKHTTP and the llm_eater's `httpPost()` return empty string on curl failure. Callers check `if (!response.empty())` — no retry, no logging of which URL failed, no distinction between "server returned empty body" and "DNS resolution failed." When all HTTP calls fail, `MKFactVerifier` produces `confidence = 0.0` and the honesty engine refuses to answer (fail-safe), but the user gets no indication the issue is network connectivity rather than genuinely missing information.

## Freshness categorization ambiguity

`MKFreshnessManager::detectCategory()` classifies topics by scanning for hardcoded keyword substrings in a fixed if-chain order. Because Technology keywords ("computer", "software", "ai") are checked before Science keywords ("physics", "chemistry"), a topic like "computer science" gets the Technology 90-day expiry rather than Science's 365 days. The first-match-wins logic means category assignment depends on keyword ordering, not signal strength.

Expiry periods differ dramatically (Politics=7 days vs Science=365 days), so misclassification causes either premature re-verification churn or stale-fact retention. Topics that match no keyword at all fall through to General (180 days), which will be the common case for many domain-specific queries.
