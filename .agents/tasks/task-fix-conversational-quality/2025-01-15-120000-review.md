# Math routing, acknowledgment context, and personality simplification

This change addresses three conversational failures: arithmetic queries like "what is 1+1" falling through to the knowledge graph (returning "I don't have that info"), acknowledgments like "yea" triggering unrelated graph lookups instead of continuing the previous topic, and an overly aggressive slang personality that buried answers under filler. The fix adds a math detection stage early in the routing pipeline, rewires the VAGUE_RESPONSE handler to reference the last topic, removes "what is" from graph keywords, and strips the personality templates down to plain conversational English.

**Watch for:** The `hasArithmeticPattern` minus-sign detection will false-positive on hyphenated numbers in prose (confirmed). The "what is" keyword removal from the graph router may cause legitimate factual queries like "what is gravity" to miss the GRAPH route entirely, falling through to the LLM instead (likely). The VAGUE_RESPONSE handler has no fallback when `lastTopic` contains user-controlled content that could produce awkward concatenations (confirmed).

## High-level view

The routing pipeline in `mk_entry.cpp` now has a math solver checkpoint between the idea engine and the conversation mode. Anything matching `isMathQuery()` — which now includes a digit-operator-digit pattern scan — gets evaluated locally and short-circuits before the smart router ever sees it. The pattern matcher's handling of the minus operator creates a false-positive risk for hyphenated content.

Acknowledgment routing works by checking whether `lastTopic` is populated in the conversation mode's topic stack. When it is, the response now explicitly names the topic ("want me to tell you more about X?") rather than firing a generic slang acknowledgment. The topic string is interpolated raw into response templates without length or format normalization.

The "what is" prefix was removed from `graphKeywords` in the smart router to prevent arithmetic from routing to GRAPH. This is a broader fix than necessary: the math solver already runs before the router, so arithmetic would be caught regardless. The removal also prevents legitimate factual queries ("what is gravity") from reaching the knowledge graph unless another keyword matches.

The personality overhaul replaces ~300 lines of slang-heavy templates with ~80 lines of natural responses. The `response_style.cpp` confidence phrases are mostly empty strings now, making `format_response()` a near pass-through.

## Arithmetic detection and the minus-sign problem

The new `hasArithmeticPattern` method scans the input for any character matching `+`, `-`, `*`, `/` with digits on both sides (skipping whitespace). A query like "tell me about the 2019-2020 pandemic" would match: digit `9`, operator `-`, digit `2`. The method would return true, `evaluateArithmetic` would extract "2019 - 2020 = -1", and return that as the answer — completely hijacking a factual query.

The minus sign doubles as a hyphen in natural language (date ranges, compound numbers, identifiers). Plus, multiply, and divide don't have this problem. A mitigation: check that the arithmetic expression constitutes the bulk of the input (after stripping "what is"), not just a substring within a longer sentence.

## Factual query regression from "what is" removal

The smart router's `graphKeywords` previously included "what is", catching queries like "what is python", "what is a black hole", "what is the EU". The math solver now runs *before* the smart router in the pipeline, so arithmetic would be caught regardless — the "what is" keyword could have stayed in `graphKeywords` without causing the original bug, as long as the math check in `mk_entry.cpp` returns early before the router is reached.

The test suite reflects this regression: the router test for "what is python" was changed to "define python", meaning the test no longer validates that natural "what is X" factual queries route correctly. These queries will now fall through to the conversation mode or LLM rather than the knowledge graph.

## VAGUE_RESPONSE topic interpolation

The implementation creates five template strings that embed `lastTopic` directly:

```cpp
"got it. want me to tell you more about " + lastTopic + "?",
"yeah, " + lastTopic + " is interesting. want to go deeper?",
```

The `lastTopic` value comes from `pushTopic()`, called with whatever subject string was extracted from the previous query. There's no length cap or normalization. If the previous interaction pushed a topic like "the difference between TCP and UDP", the response becomes "yeah, the difference between TCP and UDP is interesting. want to go deeper?" — grammatically awkward. If the topic is a full sentence accidentally stored (dependent on how `extractOpinionSubject` works upstream), it degrades further.
