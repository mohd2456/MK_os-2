# LLM-first conversation pipeline replacing Genesis waterfall

MK's conversation path has been rewired from a four-engine waterfall (Genesis -> Prometheus -> CXN -> MCE -> templates) to an LLM-first architecture: try local LLM, then cloud LLM with RAG context from the knowledge graph, then Genesis as offline fallback, then templates as last resort. The same logic is mirrored into the Telegram `generate_ai_response` handler. A new `getContextString()` on `MKBrainMemory` surfaces recent dialog turns for the LLM prompt. Five unit tests validate LLM initialization, fallback-to-empty behavior, RAG assembly, and memory formatting.

**Watch for:** Personality split between the local-LLM prompt and `MKCloudLLM::generateWithContext`'s built-in system prompt (confirmed). Missing template fallback in the Telegram path creates an asymmetry with the CLI path (confirmed). Duplicated ~50-line prompt-construction block across two call sites invites future drift (confirmed).

## High-level view

The new pipeline checks `llmEngine.isAvailable() || cloudLLM.isAvailable()` as a gate. When true, it gathers graph facts via `graph.getAll(input)`, pulls conversation history via `brainMemory.getContextString()`, and tries local then cloud generation. When both produce empty (or are unavailable), Genesis takes over as offline fallback, and templates are the final safety net. This is a clean priority inversion — Genesis drops from first to third.

The local-LLM path manually constructs a prompt embedding the system persona, facts, and history into a single string. The cloud path delegates prompt construction to `MKCloudLLM::generateWithContext`, which carries its own hardcoded system prompt. These two personas diverge in tone and instruction, meaning a user talking to MK will get a different personality depending on whether the local server or cloud API answers.

The Telegram handler (`generate_ai_response`) reproduces the full LLM-first logic but omits the template fallback that the CLI path includes, and also skips the evolution engine feeds (`prometheus.absorb`, `genesis.absorb`, `consciousnessEngine.absorb`). The Telegram LLM path is a stripped-down copy that will drift from the CLI path over time.

The prompt-construction block (system prompt + facts + history + user input) is copy-pasted verbatim between `handle_natural_query` and `generate_ai_response`. Any future change to the prompt structure requires parallel edits in two places.

## Personality divergence between local and cloud LLM paths

The local-LLM path in `mk_entry.cpp` defines:

```
"You are MK, a personal AI assistant. You are helpful, direct, and friendly.
You talk naturally like a knowledgeable friend. You are loyal to your creator Mohammed.
Keep responses concise and genuine. If you do not know something, say so honestly."
```

The cloud path delegates to `cloudLLM.generateWithContext()`, which internally builds:

```
"You are MK, a living AI. Casual, smart, loyal to Mohammed.
Use the FACTS below to answer accurately..."
```

The local prompt emphasizes "helpful, direct, friendly" with no mention of facts usage. The cloud prompt emphasizes "casual, smart" and explicitly instructs fact usage. A user whose session starts on local LLM and falls over to cloud mid-conversation will experience a personality shift. The cloud LLM's `generate()` method has yet a third persona ("Gen-Z slang, chill vibes"), though that path isn't reached by the current code since `generateWithContext` is always called.

The `generateWithContext` method doesn't expose a `systemPrompt` override the way `generate()` does. To unify behavior, either add a system-prompt parameter to `generateWithContext`, or have the caller use `generate()` directly with a pre-built prompt (like the local path does).

## Telegram handler asymmetry

In `handle_natural_query` (CLI), the fallback chain is: LLM -> Genesis -> template -> (fall through to router). In `generate_ai_response` (Telegram), it's: LLM -> Genesis -> (fall through to router). The template fallback is absent. A Telegram user in a conversational context where LLM and Genesis both fail will get a response from the smart-router's GRAPH/REASONING/INSTANT path rather than a casual conversational template.

The Telegram handler also skips `prometheus.absorb()`, `genesis.absorb()`, and `consciousnessEngine.absorb()` for successful LLM responses — it only records to `brainMemory`, `memory`, and `factExtractor`. The CLI path absorbs into all five subsystems. Prometheus/Genesis/Consciousness won't learn from Telegram conversations that go through the LLM path, creating a silent training gap between channels.

## Prompt construction duplication

The following block appears twice, character-for-character identical:

```cpp
std::string systemPrompt = "You are MK, a personal AI assistant...";
std::string fullPrompt = systemPrompt + "\n\n";
if (!relevantFacts.empty()) {
    fullPrompt += "Known facts:\n";
    for (const auto& f : relevantFacts) fullPrompt += "- " + f + "\n";
    fullPrompt += "\n";
}
if (!history.empty()) fullPrompt += "Recent conversation:\n" + history + "\n";
fullPrompt += "User: " + input + "\nMK:";
```

A static helper like `buildLLMPrompt(input, relevantFacts, history)` would eliminate the duplication and make the prompt a single source of truth.

## Test coverage gaps

Not tested: the actual LLM response path when a provider is available (would require mocking HTTP), the fallback chain ordering (local before cloud before Genesis), the Telegram handler's LLM path, prompt construction correctness, or the personality of the system prompt reaching the LLM. The fallback ordering is tested implicitly by verifying both engines return empty when unavailable, but there's no test asserting that local is attempted before cloud.
