# Mega-upgrade: three AI generation engines, expanded knowledge, and polished UX

MK OS gains three new response generation subsystems (Prometheus fluid intelligence, CXN Crystal Network, MCE Consciousness Engine), a creative Idea Engine, ~2000 new knowledge facts across six domains, an overhauled REPL with animations and session stats, Telegram bot improvements (rate limiting, retries, HTML mode, inline keyboards), an upgraded PC helper with system info and safe command execution, and an installer that now supports macOS. The existing math solver and code runner are significantly extended (calculus, statistics, JS/Rust support, input sanitization). All 12 prior compiler warnings are resolved.

**Watch for:**
- Shell injection in BirthChamber's `getRunCmd` where user-influenced `input` is interpolated into a shell `echo '...'` command with no escaping (confirmed).
- TLS certificate verification disabled (`CURLOPT_SSL_VERIFYPEER, 0L`) in Telegram plugin's HTTP requests (confirmed).
- PC helper `killProcess` skips the allowlist check entirely, letting any user-requested PID >1000 be killed without consent flow (confirmed).
- Boot initialization copies up to 500+ knowledge facts and all casual responses into multiple engines (Prometheus, CXN, MCE) synchronously, creating a cold-start latency proportional to knowledge base size (likely).
- `code_runner.cpp` sanitization blocklist can be trivially bypassed via URL-encoded or base64 payloads, or by spelling variants not in the list (confirmed: the blocklist is purely substring-matching on literal strings like `"rm -rf /"`).

---

## High-level view

The change introduces a four-tier response generation fallback chain: Prometheus -> CXN -> MCE -> template. Each engine is a standalone C++ class with its own persistence (file-based save/load) and its own bootstrap path from casual responses + knowledge facts. The design deliberately avoids external LLM calls, keeping everything rule-based and local. The engines are conceptually interesting — Prometheus uses an evolutionary "droplet pool" with fitness and mutation, CXN uses crystal pattern matching with depth tiers, MCE uses a word graph with echo memory — but all three currently produce output from the same bootstrap data (casual response strings and knowledge facts), so the diversity benefit over the template system depends entirely on runtime evolution from user interactions.

The security posture is uneven. The code runner and PC helper gain solid allowlist/blocklist defenses, but the BirthChamber (Prometheus's code evolution component) passes unsanitized strings into shell commands. The Telegram plugin disables TLS verification, which is flagged in the existing code comment as intentional for "outdated local chains" but remains a MITM vector. The killProcess function bypasses the allowlist entirely.

The boot sequence initializes all engines sequentially, each iterating through the full knowledge graph and casual responses. This is O(facts * engines) work before the first prompt. The knowledge base grew from ~7400 to ~9000 facts, making this proportionally heavier.

Test coverage remains at 7 tests / 62 assertions. None of the three new engines, the idea engine, the expanded math solver, the new Telegram commands, or the PC helper system info functions have test coverage.

1. **BirthChamber shell injection** — `getRunCmd` interpolates the `input` parameter directly into `echo '...' | timeout ...` without escaping single quotes or other shell metacharacters. Sanitize or hex-encode the input before passing to shell.
2. **TLS verification disabled** — Telegram plugin sets `CURLOPT_SSL_VERIFYPEER, 0L`, allowing MITM on all bot API traffic. Enable verification or at minimum make it configurable via env var.
3. **killProcess bypasses allowlist** — `MKPCHelper::killProcess` only checks `pid > 1000` but does not go through the command allowlist or blocked-pattern checks. A user who reaches this path can kill arbitrary non-system processes.
4. **Code runner sanitization bypassable** — The blocklist matches literal substrings (`"rm -rf /"`) but misses trivial variants like `rm -r -f /`, `rm --recursive --force /`, backtick injection, `$(...)` command substitution, or newline-separated commands in bash.
5. **Cold-start latency** — Three engines each iterate up to 500 knowledge facts + all casual responses during boot, with O(n) copies and string allocations. For 9000+ facts this will grow.
6. **No test coverage for new subsystems** — Prometheus, CXN, MCE, Idea Engine, expanded math solver (calculus/stats/base-convert), Telegram commands, and PC helper are all untested.
7. **Rate-limit busy wait** — Telegram retry uses a spin loop (`while (elapsed < delay) {}`) which burns CPU. Use `std::this_thread::sleep_for` instead.
8. **Duplicate bootstrap code** — Steps 4c, 4d, 4e in `mk_entry.cpp` each independently collect casual responses into local vectors using near-identical lambda patterns, tripling the allocation for the same data.
9. **Dead temperature scheduling** — `currentTemperature_` in FluidDynamics is initialized to 0.7f but never updated by any caller. The temperature scheduling described in comments is not wired up. Either implement the scheduler or remove the dead code path.
10. **Response scoring penalizes valid patterns** — `scoreResponse()` uses `uniqueWords/totalWords` as quality proxy, which penalizes intentional repetition (emphasis, lists) and rewards unrelated word salad equally. Consider context-aware scoring.

## BirthChamber shell injection vector

The `MKBirthChamber::getRunCmd` method builds shell commands like:

```cpp
return "echo '" + input + "' | timeout 5 " + outPath + " 2>/dev/null";
```

The `input` parameter flows from `conceive()` which builds test inputs from goal droplet content — ultimately derived from user natural language input that was absorbed into the Prometheus pool. A user who types `'; rm -rf / #` as a query would eventually have that string stored in a droplet's content. When that droplet is later selected as a "goal" during code evolution, the unescaped content reaches this shell command. The `code_runner.cpp` has its own `isDangerous()` check, but `BirthChamber` performs no such validation — it uses raw `std::system()` for compilation and `execCapture(popen(...))` for execution without any sanitization layer.

The fix is straightforward: either route through `MKCodeRunner` (which already has `isDangerous()`), or at minimum escape single quotes in the input string before interpolation.

## Telegram TLS and bot token handling

The Telegram plugin explicitly disables TLS verification:

```cpp
curl_easy_setopt(curl, CURLOPT_SSL_VERIFYPEER, 0L);
```

This was present in the original code, but the diff modifies the `request()` method substantially (adding retry logic, POST support, timeouts) while preserving this setting. Given that the bot token is transmitted in the URL path of every request, disabling peer verification means any network-level attacker can intercept the token.

The retry mechanism uses a spin-loop busy wait:

```cpp
while (std::chrono::duration_cast<std::chrono::milliseconds>(
    std::chrono::steady_clock::now() - start).count() < RETRY_DELAY_MS * (attempt + 1)) {
    // wait
}
```

This burns 100% CPU for up to 3 seconds (3 retries * 1s each with linear backoff). `<thread>` is already included elsewhere in the project — `std::this_thread::sleep_for` would be appropriate.

## PC helper allowlist gap in killProcess

`killProcess(int pid)` only checks `pid > 1000` and then directly invokes `std::system("kill " + std::to_string(pid))`. Since PID is an integer, there's no injection risk, but the function bypasses the entire safety model: if the caller exposes this to user input (e.g., a future `/kill <pid>` command), any user process can be terminated without the allowlist gating. The kill command itself is not in the `getAllowedCommands()` set, yet `killProcess` executes it anyway.

## Code runner sanitization limits

The `isDangerous()` blocklist in `code_runner.cpp` matches exact substrings like `"rm -rf /"`. This catches the literal string but not semantic equivalents:

- `rm -r -f /` (split flags)
- `rm --recursive --force /` (long flags)
- `` `rm -rf /` `` (backtick execution inside otherwise-safe code)
- `$(rm -rf /)` (command substitution in bash)
- Python's `os.system("rm -rf /")` (not caught because the check is on the code string, not execution intent)

The design is defense-in-depth (timeouts, `/tmp` paths, output limits), so the blocklist provides weaker guarantees than it appears to but isn't the sole safeguard.

## Boot sequence and memory layout

The initialization in `main()` follows this pattern three times:

```
collect casualTexts into local vector (one addCasual per category)
collect knowledgeFacts from graph (up to 300-500 edges)
call engine.initialize(casualTexts, knowledgeFacts, ...)
```

Each engine receives its own copy of the same data. With 602 casual responses and 500 knowledge facts, that's ~3x1100 string copies during boot. The CXN engine caps at 300 facts, Prometheus at 500, and MCE at 500 — three separate iterations over `graph.getAllEdges()`. A shared bootstrap data structure passed by const reference would eliminate the duplication.

## Prometheus temperature scheduling and response scoring

The `currentTemperature_` field in `MKFluidDynamics` is initialized to 0.7f. It has a setter (`setTemperature`) and affects resonance scoring via a noise injection path (`if (currentTemperature_ > 0.7f) { ... noise ... }`). But nothing in the diff calls the setter — the temperature never changes. The "temperature scheduling (start hot for brainstorming, cool down for precision)" described in FEAT-002 is not wired up.

The `scoreResponse()` function uses `uniqueWords / totalWords` as a quality proxy. This metric treats "no no no, that's wrong" (intentional repetition for emphasis) the same as a broken repetition bug, and treats a sentence with many unrelated words as high-quality regardless of coherence. For the short template-style responses that dominate the bootstrap pool, most will score similarly, reducing the selection pressure that the multi-strategy assembly is designed to provide.

## Test coverage gap

The diff adds ~10,000 lines of new behavioral code. None of the new subsystems have test coverage. The math solver extensions (pure functions with deterministic output) and the code runner sanitization (can be tested with known-dangerous inputs) would be particularly straightforward to test and are the most likely to contain parsing bugs given the fragile `string::find` + `std::stod` extraction patterns used throughout.
