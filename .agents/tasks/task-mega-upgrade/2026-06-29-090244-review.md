# Mega-upgrade v2: security fixes, temperature scheduling, and new tests

The fix commit addresses the most critical issues from the v1 review. Shell injection in BirthChamber is closed via single-quote escaping. TLS verification is re-enabled in the Telegram plugin. The killProcess function now routes through `isCommandSafe()`. The busy-wait spin loop is replaced with `sleep_for`. Temperature scheduling is wired into Prometheus via keyword detection. Duplicate bootstrap vectors are consolidated into a shared collection. Code runner sanitization is expanded with split-flag, command-substitution, and scripting-language patterns. Two new test suites cover math solver operations and code runner sanitization (9 tests, 102 assertions total, all passing).

**Watch for:**
- Temperature scheduling is immediately overwritten: `setTemperature()` is called before `resonate()`, but `resonate()` internally calls `updateTemperatureSchedule()` which overwrites the value based on generation count (confirmed).
- BirthChamber executes assembled code without passing through `isDangerous()` — absorbed response droplets at precision >= 0.7 feed back as "code" droplets and can be executed (confirmed).
- Code runner blocklist includes `exec(` and `eval(` which blocks all Python code using those builtins, including legitimate `eval("2+2")` patterns (confirmed, likely intentional but over-broad).
- Three new generation engines (MCE, CXN, Prometheus), the idea engine, conversation mode, expanded math solver, Telegram commands, and PC helper remain untested (confirmed).

## High-level view

The temperature scheduling fix has a sequencing bug: the keyword-based temperature override is applied before `resonate()` is called, but `resonate()` increments the generation counter and calls `updateTemperatureSchedule()` unconditionally, overwriting whatever temperature was just set. The schedule always wins. The fix is either to apply the keyword override after `resonate()` returns, or to skip `updateTemperatureSchedule()` when an explicit override was set.

BirthChamber's shell escape correctly neutralizes the `input` parameter. However, BirthChamber assembles and executes code fragments without any `isDangerous()` gating. The `absorb()` method adds MK's own responses at precision 0.7-0.8 when they contain code markers (backticks, `def`). These droplets satisfy `isCode()` and can be selected by `gatherCodeDroplets()` in future invocations, then assembled and executed. The timeout (5 seconds) and /tmp sandbox provide defense-in-depth, but the code path lacks the blocklist check that protects the user-facing code runner.

The broad `exec(` and `eval(` patterns in the code runner blocklist will produce false positives for legitimate user code using those Python builtins. Given the defensive posture this is an acceptable trade-off, flagged as a known limitation rather than a blocking concern.

MCE, CXN, Prometheus, IdeaEngine, ConversationMode, Telegram commands, and PC helper have no test coverage. The math solver tests demonstrate the pattern for extending coverage to deterministic subsystems like CXN crystal lookup or MCE word graph generation.

## Temperature scheduling sequencing bug

The `generate()` method in `prometheus_engine.cpp` sets temperature based on keyword detection:

```cpp
if (needsPrecision) {
    dynamics_.setTemperature(0.3f);
} else if (needsCreativity) {
    dynamics_.setTemperature(0.9f);
}

// Step 1: RESONATE
auto vibrating = dynamics_.resonate(input, pool_);
```

Inside `resonate()` (fluid_dynamics.cpp line 204-205):

```cpp
generationCount_++;
updateTemperatureSchedule();
```

`updateTemperatureSchedule()` unconditionally sets temperature based on generation count (0.9 for first 5 calls, 0.7 for 5-15, 0.5 for 15-30, 0.3 after 30). The keyword-based override is immediately lost. After generation 30, temperature is permanently 0.3f regardless of whether the user asked a creative question. Fix: move the keyword override to after `resonate()` returns, or add a `bool explicitOverride_` flag that `updateTemperatureSchedule()` respects.

## BirthChamber code execution without sanitization

The execution path for evolved code:

```
absorb(input, response, continued)
  -> pool_.addDroplet(response.substr(0,200), 0.8f, ..., "generated")
    -> isCode() returns true (precision >= 0.7)

[future call]
generate(input)
  -> gatherCodeDroplets(vibrating)
    -> picks all vibrating droplets where isCode() == true
  -> chamber_.conceive(goalIds, codeIds, pool_)
    -> assembleCandidate(fragments, goal, variation)
  -> chamber_.tournament(candidates, tests)
    -> getRunCmd() -> execCapture(popen(cmd))
```

No `isDangerous()` check sits between assembly and execution. Risk is bounded by: (1) responses are MK's own output, not raw user text; (2) only the first 200 chars of a response are absorbed; (3) the 5-second timeout limits damage. If a user crafts input that causes MK to generate a response containing dangerous patterns, those patterns could execute via this feedback loop. Adding `isDangerous()` to the assembled candidate before execution would close the gap.

## Code runner remaining bypass vectors

The expanded blocklist is substantially improved but remains a substring matcher. Remaining bypass paths (low priority given timeout/sandbox defense-in-depth):
- Multi-line variable concatenation: `a="rm"; b="-rf"; $a $b /`
- Language-specific execution not in the list (Perl backticks, Ruby `system`)
- Unicode homoglyphs or zero-width characters in identifiers

The 5-second timeout and /tmp working directory are the actual security boundaries; the blocklist provides early rejection of obvious attacks.
