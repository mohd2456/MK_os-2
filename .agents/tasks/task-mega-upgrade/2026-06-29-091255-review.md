# Mega-upgrade v3: final review

MK OS gains three response generation engines (Prometheus, CXN, MCE), an idea engine, ~2700 new knowledge facts, overhauled REPL with animations, Telegram bot hardening, PC helper expansion, macOS installer support, and significant expansions to the math solver and code runner. The build is clean under `-Wall -Wextra -Werror`, and the test suite has grown from 7/62 to 11/160 (tests/assertions). All critical issues from prior reviews are resolved.

**Watch for:**
- BirthChamber's `containsDangerousPattern()` is a strict subset of CodeRunner's `isDangerous()` — missing `exec(`, `eval(`, `wget http`, `curl http`, `mv /* `, `execsync(`, `spawnsync(` (confirmed). Low severity given /tmp sandbox and 5s timeout.
- Prometheus, CXN, and MCE engines have zero test coverage (confirmed). The new tests cover deterministic subsystems (math solver, sanitization, conversation mode, idea engine) but not the three generation engines that form the core behavioral addition.

**Verdict**: APPROVED

## High-level view

Prior security concerns are closed: shell escape on BirthChamber input, TLS re-enabled, killProcess gated through `isCommandSafe()`, spin-loop replaced with `sleep_for`, and `containsDangerousPattern()` preventing the absorb-feedback loop from executing dangerous assembled code. The remaining gap — BirthChamber's blocklist being a subset of CodeRunner's — is bounded by defense-in-depth (5s timeout, /tmp sandbox, assembled code originates from MK's own output not raw user input).

The temperature scheduling override is now applied *after* `resonate()`, which means it persists into the next `generate()` call rather than being immediately overwritten. The practical effect is a one-turn lag: the first response to "write me creative poetry" uses the schedule temperature; subsequent creative queries use the hot override. `temperatureAdjust()` uses `identity_.energy` rather than `currentTemperature_`, so the output mutation path is unaffected by this lag.

Test coverage now exercises math solver operations, code runner sanitization (including split-flag and substitution patterns), conversation mode classification, and idea engine generation. The three generation engines are the primary coverage gap — their deterministic subcomponents (FluidDynamics scoring, CrystalStore trigger lookup, WordWeaver graph walk with fixed seed) could be exercised without mocking randomness.

## BirthChamber blocklist subset gap

`containsDangerousPattern()` in `birth_chamber.cpp` contains 28 patterns. `isDangerous()` in `code_runner.cpp` contains 38+ patterns. The delta:

```
"exec(", "eval(", "os.exec",
"wget http", "curl http",
"mv /* ", "/dev/null >", ">/dev/sda",
"$(reboot", "`reboot",
"execsync(", "spawnsync("
```

Since BirthChamber assembles Python and bash fragments from `code_fragments.cpp` and executes them directly via `popen()` (not through CodeRunner), the missing `exec(` and `eval(` patterns are the most relevant gap — an assembled Python fragment using `eval()` would pass BirthChamber's check with no second gate.

The exploitation path is narrow: a user would need to craft input that causes MK to generate a response containing `eval(...)`, which gets absorbed at precision 0.8, selected in a future tournament, assembled into a candidate, and executed. The 5-second timeout caps damage. The fix is to extract the pattern list into a shared definition.

## Temperature override one-turn lag

```cpp
auto vibrating = dynamics_.resonate(input, pool_);  // increments gen, calls updateTemperatureSchedule()

// Override AFTER resonate — persists for next call
if (needsPrecision) dynamics_.setTemperature(0.3f);
else if (needsCreativity) dynamics_.setTemperature(0.9f);
```

`currentTemperature_` only affects noise injection *inside* `resonate()`. Since `resonate()` already ran, the override takes effect on the *next* `generate()` call. This one-turn lag means the system converges after one exchange rather than responding immediately to context switches.
