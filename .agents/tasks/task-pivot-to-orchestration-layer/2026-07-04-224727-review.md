# Pivot to orchestration layer: single-LLM call, tool framework, token efficiency, personal memory

MK OS pivots from a multi-engine AI system (Genesis, Prometheus, CXN, MCE, thinking engine, decision engine, smart router, composer, conversation mode, neural net, idea engine â€” ~12,500 lines) to a clean orchestration layer built around a single LLM call per user message. The new architecture introduces four components: `MKOrchestrator` (single entry point for all natural language), `MKToolRegistry`/`MKToolExecutor` (9-tool framework with safety guards), `MKContextBuilder` (progressive token-budget management), and an upgraded personal memory system (SQLite-backed long-term memory with fact auto-persistence). The deleted engines are replaced by ~1,700 lines of new code that delegates all reasoning to the external LLM while MK controls context, tools, and knowledge.

**Watch for:**
1. **write_file tool path traversal via relative path** (confirmed) â€” `isWritePathAllowed` passes any relative path without literal `..`, but doesn't resolve to canonical absolute paths. A path using symlinks or CWD manipulation bypasses the check. Additionally, `path.find("/tmp") == 0` matches `/tmpevil/payload`.
2. **Tool call JSON parsing fails on nested objects and arrays** (confirmed) â€” `parseArgsObject` handles `"key": "value"` pairs but any JSON value containing a nested `{` or `[` produces incorrect parsing because the scanner finds `"` inside nested structures and treats them as key boundaries.
3. **SQLite null dereference on failed column read** (likely) â€” `loadFromDisk` and `queryRelevantMemories` cast `sqlite3_column_text()` directly to `const char*` without null-checking. If a column is NULL (corrupted DB), this is undefined behavior.
4. **Context builder budget bypassed for cloud LLM calls** (confirmed) â€” `callLLM()` constructs its own `enhancedPrompt = *systemPrompt + toolRegistry->buildToolPrompt()` for the cloud path, ignoring whatever `MKContextBuilder` decided to drop. Token-efficiency progressive dropping only applies to local LLM.
5. **Thread-safety gap on CLI path** (likely) â€” `handle_natural_query` doesn't acquire `systemMutex` before calling the orchestrator. The Telegram polling loop does. Concurrent access races on `brainMemory`, `learningEngine`, and `factExtractor`.

## High-level view

The entire multi-engine routing stack is gone. All queries flow through `MKOrchestrator::respond()` which gathers knowledge-graph facts and personal facts, builds one token-optimized prompt via `MKContextBuilder`, makes one LLM call (local first, cloud fallback), detects tool-call JSON in the response, executes up to 3 tools per turn, and feeds results back through a synthesis call.

The tool framework registers 9 tools with JSON schemas and a prompt that teaches the LLM to emit `{"tool": "...", "args": {...}}`. The executor dispatches to SSH, Docker, shell, file I/O, web APIs, device registry, and learning engine with safety guards. The write-path allowlist has a gap around relative paths, `/tmp` prefix matching, and symlinks.

Token efficiency is managed by `MKContextBuilder` which scores facts by keyword overlap, compresses history (last 2 turns verbatim, older summarized as topic keywords), and progressively drops content if the prompt exceeds 1500 tokens. However, the cloud LLM path constructs its own prompt independently, partially bypassing this budget system.

Personal memory is two-tiered: short-term (6-turn sliding window) and SQLite long-term. When short-term fills, the oldest 2 turns are archived with an importance score. The fact extractor (now 56 rules) auto-persists to the learning engine. Personal facts are injected into every LLM prompt.

## write_file path traversal gap

```cpp
static bool isWritePathAllowed(const std::string& path) {
    if (path.find("/tmp/") == 0 || path.find("/tmp") == 0) return true;
    // ...
    if (path.find("..") != std::string::npos) return false;
    if (path[0] != '/') return true; // relative path, assumed within MK_OS
    return false;
}
```

Three distinct problems:

1. `path.find("/tmp") == 0` matches any path starting with `/tmp` â€” including `/tmpevil/payload` or `/tmp_malicious`. Should be `/tmp/` with trailing slash.
2. The comment says "relative path, assumed within MK_OS" but there's no enforcement that the working directory is MK_OS/ at runtime. Relative writes resolve against wherever MK was launched from.
3. `path.find("..") != std::string::npos` is a substring check, not a path-component check. A legitimate directory `something..else` is incorrectly blocked, while symlink traversal passes silently.

For a tool exposed to LLM-generated content (the LLM constructs the path string), this needs `realpath()` resolution followed by prefix containment.

## Tool-call JSON parsing limitations

`MKToolRegistry::parseToolCall()` searches for `{"tool":` in the LLM response, then uses custom string-scanning to extract the tool name and args. The args parser (`parseArgsObject`) iterates through the JSON finding `"key"` then skipping to `":"` then reading the value.

If a value contains a nested array like `"items": ["a", "b"]`, the parser finds `"a"` after the `[` and interprets it as the next key, producing garbage. Escaped quotes within string values are handled correctly by the escape loop, so `"object": "foo\"bar"` works. The failure mode is specifically unescaped nested structures.

Currently all 9 registered tools use flat `string â†’ string` args, so this doesn't bite today. But extending a tool to accept a list or nested object will silently produce wrong results with no error.

## Cloud LLM bypasses the token budget

In `MKOrchestrator::callLLM()`:

```cpp
if (cloudLLM && cloudLLM->isAvailable()) {
    std::string enhancedPrompt;
    if (systemPrompt) {
        enhancedPrompt = *systemPrompt;
        if (toolRegistry && config.enableToolCalls) {
            enhancedPrompt += toolRegistry->buildToolPrompt();
        }
    }
    response = cloudLLM->generateWithContext(input, facts, history, "", enhancedPrompt);
}
```

This constructs its own prompt independently of what `MKContextBuilder` decided. If the builder dropped the tool prompt or switched to the minimal system prompt to stay within budget, the cloud path ignores that decision and sends the full system prompt plus the full tool prompt. The token-efficiency system is effectively local-LLM-only.

The fix is straightforward: pass the already-assembled `prompt` string (which the builder produced) to the cloud path, or at minimum pass `result.usedMinimalPrompt` / `result.factsIncluded` state forward so the cloud path can respect the same budget decisions.

## SQLite null-pointer risk

```cpp
mem.topic = (const char*)sqlite3_column_text(stmt, 1);
```

The schema declares `topic TEXT NOT NULL`, so under normal operation this column can't be NULL. But `queryRelevantMemories` uses a LIKE query â€” a corrupted database, a migration mishap, or a concurrent writer could produce NULL values. The fix is a one-line null-guard:

```cpp
const char* val = (const char*)sqlite3_column_text(stmt, 1);
mem.topic = val ? val : "";
```
