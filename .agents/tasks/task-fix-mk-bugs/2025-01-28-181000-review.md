# Prompt-leak sanitizer, command wiring, and thinking engine validation (v2)

The v1 review raised five issues. Three have been addressed: HTML escaping now uses an unconditional `escapeHtml()` applied solely to LLM-generated replies in the poll loop (issue 1), entity strings removed from `safeTags` (issue 3), and the sanitizer's system-prompt similarity check now references the actual `MK_SYSTEM_PROMPT` constant (issue 4). Issues 2 (CLI empty-response propagation) and 5 (echo detection edge case for 6-9 char inputs) were accepted as low severity.

**Watch for:**
1. **Dead code left in telegram.cpp** (confirmed) ‚Äî `escapeForUser()` and `shouldEscapeHtml()` are no longer called from anywhere. The active escaping path is the static `escapeHtml()` in mk_entry.cpp. The dead methods still reference the now-removed-entity-containing `safeTags` logic that was the original injection vector.
2. **Unconditional escaping destroys legitimate formatting from `generate_ai_response` sub-paths** (possible) ‚Äî if any future code path returns pre-formatted HTML through `generate_ai_response`, the blanket `escapeHtml` will mangle it. Currently all such paths return plain text, so this is a latent concern, not a live bug.

**Verdict**: APPROVED

## High-level view

The poll loop's reply path exclusively handles LLM-generated text, and all MK-formatted messages (commands like `/setkey`, `/status`, `/help`) go through separate `sendMessage` calls earlier in the dispatch chain. Applying `escapeHtml` unconditionally at the single LLM reply site eliminates the injection vector without risking double-escape of MK's own messages.

The sanitizer now references `MK_SYSTEM_PROMPT` directly for its similarity check, eliminating the drift risk between the detection logic and the actual prompt.

`escapeForUser` and `shouldEscapeHtml` in telegram.cpp are now dead code ‚Äî unreachable from any call site. They carry the vestige of the bypassed tag-detection logic.

## HTML escaping architecture after v2 fix

The v1 design delegated escaping to `sendMessage` via `escapeForUser`, which consulted `shouldEscapeHtml` ‚Äî a tag-presence heuristic that could be tricked by LLM output containing literal HTML tag names. The v2 fix sidesteps this entirely:

```
telegram_poll_loop
  ‚îî‚îÄ reply = generate_ai_response(...)   ‚Üê always plain text
  ‚îî‚îÄ if (reply.empty()) reply = fallback  ‚Üê plain text literal
  ‚îî‚îÄ reply = escapeHtml(reply)            ‚Üê unconditional, no heuristic
  ‚îî‚îÄ sendMessage(chatId, reply)           ‚Üê sendMessage is now a pass-through
```

MK's own HTML messages (`/setkey`, `/setmodel`, `/providers`, `/status`, `/help`, keyboard messages) are sent via separate `sendMessage` calls in the command dispatch block (lines 2024-2122) that never pass through `escapeHtml`. The separation is structural ‚Äî commands hit `if/else if` branches that `sendMessage` directly and then `continue`, while the LLM reply path lives in the final `else` branch.

## Dead code: escapeForUser / shouldEscapeHtml

`escapeForUser` and `shouldEscapeHtml` on `MKTelegram` are not called from mk_entry.cpp or anywhere else. They represent the original (flawed) design that the v2 fix replaced. Leaving them risks a future contributor calling `escapeForUser` thinking it's the intended API, which would reintroduce the tag-detection bypass.
