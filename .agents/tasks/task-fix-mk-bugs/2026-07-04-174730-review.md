# Natural language math, greeting bypass, provider health, and smart router word boundaries

Six behavioral fixes to the MK OS C++ AI assistant: natural language math detection ("5 times 3" → 15), an early-return path that handles greetings without touching the LLM stack, a thinking engine short-circuit for trivial inputs, honest low-confidence phrasing, a provider health check that prefers Groq over a dead OpenRouter, and word-boundary enforcement in the smart router's keyword matching. Build passes, all 67 tests pass.

**Watch for:** The early greeting intercept catches `SMALL_TALK` — which `classifyInput` assigns to *any* ≤6-word input that doesn't start with a recognized question word. Short imperative requests like "tell me about docker" or "explain kubernetes" (no question prefix) would be swallowed by the casual response path instead of reaching the LLM (confirmed). The Telegram path (`generate_ai_response`) also lacks a math-solver check before the intercept, so bare math like "5 times 3" sent via Telegram gets a chill casual reply instead of 15 (confirmed).

## High-level view

The math solver gains two methods — `isNaturalLanguageMath` and `evaluateNaturalMath` — that pattern-match operator words ("times", "divided by", "plus", "minus") flanked by digits, extract the operands, and compute the result. It doesn't attempt chained operations or parenthetical expressions, which is a reasonable scope boundary for v1.

The greeting early-return in both `handle_natural_query` and `generate_ai_response` avoids the LLM entirely for inputs classified as GREETING, GOODBYE, VAGUE_RESPONSE, or SMALL_TALK. The SMALL_TALK bucket in `classifyInput` is overbroad: it captures all ≤6-word non-question inputs, including legitimate short commands and requests. This creates a regression where short imperative queries get nonsensical casual responses.

The thinking engine adds a hardcoded skip list of ~30 greeting/farewell/acknowledgment tokens plus a ≤2-word heuristic. The skip list partially duplicates the classification logic in `conversation_mode.cpp`; the two lists will drift independently.

Provider health routing introduces `isProviderHealthy()` with a `failCount > 2` threshold on top of the existing `failCount >= maxFails` (which is 5). The thinking engine's `pickThinkingProvider()` now calls `isProviderHealthy` instead of `isProviderAvailable`, making it skip providers that have accumulated 3+ consecutive failures even if they haven't hit the hard offline threshold.

The smart router's `countKeywordMatches` now enforces word boundaries (non-alphanumeric character or string edge) before and after each keyword. "time" no longer matches "times" or "sometimes".

## SMALL_TALK classification overbreadth

The `classifyInput` function in `conversation_mode.cpp` uses this cascade: greetings → goodbyes → vague responses (≤3 words) → topic changes → stories → opinions → emotions → question-word prefix → **default: if words ≤ 6, return SMALL_TALK**. The question-word prefix list is: what, who, where, when, why, how, which, is it, are there, can you, do you, does, did, will, would, could, should. Anything that doesn't start with one of these words falls through.

"tell me about docker" (4 words, starts with "tell") → SMALL_TALK. "explain kubernetes" (2 words, starts with "explain") → SMALL_TALK. "summarize this" → SMALL_TALK. "run a scan" → SMALL_TALK. All of these would be intercepted by the early-return block in both `handle_natural_query` and `generate_ai_response`, which responds with `db.pick(db.mood_chill)` — a random casual acknowledgment — instead of routing the input to the smart router, LLM, or command dispatch.

The pre-existing `FRIEND MODE` block (which follows the early intercept in `handle_natural_query`) used `isConversation(input)` which has its own more nuanced logic. The new early intercept bypasses that entirely for anything classified GREETING/GOODBYE/VAGUE_RESPONSE/SMALL_TALK.

The fix is straightforward: remove `MKInputType::SMALL_TALK` from both early intercept guards. The original FRIEND MODE block already handles conversational routing correctly; SMALL_TALK was never meant to absorb imperative commands.

## Telegram path: bare math bypass

In `handle_natural_query` (CLI path), the math solver runs *before* the greeting intercept:

```
MATH SOLVER → EARLY GREETING → FRIEND MODE → smart router → ...
```

In `generate_ai_response` (Telegram path), there's no math check at all before the intercept:

```
EARLY GREETING → isConversation → 3-layer LLM → ...
```

"what is 5 times 3" via Telegram works (starts with "what" → classified as QUESTION → not intercepted). But "5 times 3" alone is 3 words, no question prefix → SMALL_TALK → intercepted → casual reply. Either add a math check before the Telegram intercept, or (simpler) remove SMALL_TALK from the intercept types.

## Provider health threshold interaction

`isProviderAvailable` marks a provider unavailable at `failCount >= maxFails` (5). `isProviderHealthy` adds a stricter check: `failCount > 2` (i.e., 3+ failures). The two checks are:

```cpp
// isProviderAvailable:
if (p.failCount >= p.maxFails) return false;  // ≥5 → unavailable

// isProviderHealthy:
if (p.failCount >= p.maxFails) return false;  // ≥5 → unhealthy (redundant)
if (p.failCount > 2) return false;            // ≥3 → unhealthy
```

The `failCount >= maxFails` check in `isProviderHealthy` is dead code since `failCount > 2` is always triggered first (given maxFails is 5). Not a bug, but confusing to maintain. The intent — "healthy means fewer than 3 consecutive failures" — would be clearer expressed as `isProviderAvailable() && failCount <= 2`.

## Natural language math: false-positive resistance

The detection requires digits on both sides of the operator word. "I ordered 3 times already" — has a digit before " times " but no digit after, so it won't false-positive. The digit-on-both-sides requirement is a solid guard against conversational uses of "times", "plus", "minus".

"five times three" (written-out numbers) would not match. Acceptable for v1 scope; the user's stated examples all use numerals.
