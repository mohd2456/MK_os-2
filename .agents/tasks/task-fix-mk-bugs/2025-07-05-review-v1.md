# Orchestrator bug fixes: double prompting, tool-call leaks, crypto placeholders, and weather humidity

Seven fixes address the most user-visible regressions in MK OS's orchestrator layer. The core issue was that `callLLM()` and `synthesizeWithToolResult()` used `cloudLLM->generateWithContext()` which rebuilt the entire prompt from scratch — ignoring the carefully token-budgeted prompt from `MKContextBuilder`. Additional fixes expand tool-trigger keywords so time/weather queries actually get tool descriptions, harden `parseToolCall()` against markdown fences, strip raw tool JSON from responses before the user sees them, replace fake crypto portfolio placeholders with an honest "no portfolio" message, add a system prompt instruction to respect user formatting preferences, and fix weather humidity always reading 0%.

**Watch for:** The `"now"` keyword in `queryMightNeedTools()` matches as a substring inside `"know"`, `"known"`, `"snowball"`, etc. — triggering tool descriptions (and consuming token budget) for a huge fraction of ordinary messages (confirmed). The `stripToolCallJson()` function trims the entire response after stripping, which could collapse legitimate multi-paragraph whitespace (possible). 54 duplicate `mars|is_a|planet|1` entries were committed to `learned_facts.mk` — test debris that inflates the knowledge graph (confirmed).

## High-level view

The double-prompt fix is the highest-value change. `generateWithContext()` was appending facts and history on top of the already-assembled prompt, sending a bloated context to the LLM that duplicated everything. Switching to `generate(input, prompt)` sends the pre-built prompt as the system message directly, making the token budget in `MKContextBuilder` actually meaningful.

The tool-trigger keyword expansion closes the gap where queries like "what time in Fort Worth" never got tool descriptions in the prompt. However, keywords like `"now"`, `"current"`, `"what is"`, and `"check"` are so common in ordinary conversation that they'll trigger tool descriptions far more often than needed — a false-positive tax on token budget.

`parseToolCall()` now strips markdown fences before searching for JSON, and `stripToolCallJson()` acts as a post-processing safety net that removes leftover `{"tool":...}` JSON from the final response before the user sees it.

The crypto `/crypto` command replaces fake `$----.--` placeholders with "No real portfolio connected" and a pointer to `/paper`. The system prompt gains an instruction to follow explicit user formatting preferences. The weather API fix adds `&current=relative_humidity_2m` to the URL and parses the separate `"current"` JSON object for humidity data.

## Double-prompt elimination in callLLM and synthesize

The base branch's `callLLM()` called `cloudLLM->generateWithContext(input, facts, history, "", prompt)`. Inside `generateWithContext()`, the prompt parameter was used as a `systemPromptOverride`, but the method then *appended* the same facts and history again:

```cpp
// generateWithContext() internally does:
system = systemPromptOverride + "\n\n";
system += "KNOWN FACTS:\n";
for (const auto& fact : relevantFacts) system += "- " + fact + "\n";
system += "Recent conversation:\n" + conversationHistory + "\n";
return generate(userInput, system);
```

Since `prompt` (the override) already contained facts and history (assembled by `MKContextBuilder`), the LLM received everything doubled. The fix replaces both call sites with `cloudLLM->generate(input, prompt)`, which passes the pre-built prompt as the system message without re-assembly.

In `synthesizeWithToolResult()`, the base branch assembled its own prompt string (with facts, history, tool result), then passed that entire string plus the raw facts/history to `generateWithContext()` — tripling some context. The fix uses `generate(input, prompt)` with the locally assembled string, which avoids the doubling but still bypasses the token budget (Issue #4).

## Tool-call stripping pipeline

The tool-call lifecycle has three stages: (1) `parseToolCall()` finds and extracts the JSON for execution, (2) the tool loop synthesizes a natural-language response incorporating the result, (3) `stripToolCallJson()` removes any residual JSON that slipped through (e.g., when the loop hit its 3-call cap, or synthesis echoed back tool JSON).

The fenced-block stripper only removes fences whose content contains `"tool"`. One edge case: if the LLM produces a code block containing the literal string `"tool"` as part of a code example, it would be incorrectly stripped. Unlikely for this assistant's use cases but worth noting.

## queryMightNeedTools substring matching

The keyword list uses `lower.find(kw)` — pure substring matching. The most problematic entries:

- `"now"` — matches inside "know", "known", "acknowledge", "snowball" (extremely common)
- `"current"` — matches inside "currently" (common conversational word)
- `"what is"` — matches "what is your name", "what is MK" (casual identity questions)
- `"check"` — matches "unchecked", "checkout", "checker"

Every false-positive match includes tool descriptions in the prompt. With a 1500-token budget, this crowds out conversation history and knowledge graph facts. The `"now"` problem is the most severe because "know" appears in a large fraction of natural language.

The pre-existing keywords (`"temperature"`, `"search"`, etc.) are specific enough to avoid false positives. The new keywords should follow the same pattern: use phrases unlikely to appear as substrings of common words, or add leading/trailing spaces.
