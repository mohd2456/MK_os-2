# Prompt-leak sanitizer, command wiring, and thinking engine validation

This change adds three defensive layers to MK-OS: a `sanitizeLLMResponse()` function that strips leaked prompt fragments from LLM output before it reaches the user, a `validateThinkingOutput()` function in the thinking engine that rejects garbage reasoning from weak 3B models, and HTML escaping in the Telegram transport. It also wires three previously-dead commands (`/crypto`, `/devices`, `/goals`) into the Telegram dispatcher, adds an empty-reply guard, and fixes three failing tests whose assertions contradicted the decision engine's intentional design (returning empty for conversational inputs).

**Watch for:**
1. **HTML escaping bypass via LLM-injected tags** (confirmed) — any LLM response containing `<b>`, `<code>`, or similar strings disables all escaping for that response, which is an injection vector.
2. **Double-escaping of pre-escaped help text** (confirmed) — the `/help` message passes through `sendMessage` → `escapeForUser`, which checks for `&lt;` as a "safe tag" and skips escaping, but this behavior is fragile and depends on the presence of pre-escaped entities.
3. **Sanitizer returns empty without fallback in some paths** (likely) — `sanitizeLLMResponse` can return `""` in paths where no empty-reply guard exists (e.g., `synthesizeResponse`, `handle_natural_query`).

## High-level view

The prompt-leak sanitizer is a post-processing filter applied at every LLM response site. It uses pattern matching against a hardcoded list of known instruction phrases, and a chunked similarity check against the system prompt. The design is tightly coupled to exact prompt strings — any prompt rewording will need a corresponding sanitizer update, and the similarity check uses a paraphrased prompt constant rather than referencing the actual `MK_SYSTEM_PROMPT`.

The thinking engine validator catches degenerate output from small models: echo of user input, prompt regurgitation, and word-repetition loops. It returns empty on rejection, which triggers the existing fallback chain.

HTML escaping is applied unconditionally in `sendMessage`, but the decision to escape or not is delegated to `shouldEscapeHtml`, which skips escaping if the text contains any recognized HTML tag substring. This means an LLM response containing a literal `<b>` anywhere passes through entirely unescaped — a Telegram parse error or injection vector depending on what the model outputs.

The `"container"` keyword addition to the decision engine's short-input heuristic prevents "show my containers" from being misclassified as conversational and skipping tool dispatch.

## Prompt-leak sanitizer design and coupling

`sanitizeLLMResponse` in mk_entry.cpp operates in three passes: count leak-pattern hits (if ≥2, reject entirely), strip individual lines containing leak patterns, then check trailing `User:`/`MK:` lines and system-prompt similarity.

The coupling concern: the 14 leak patterns are hardcoded strings taken verbatim from the current prompt template. The function has no reference to `MK_SYSTEM_PROMPT` or to the prompt construction in `buildLLMPrompt`. If someone edits the prompt wording (e.g., changes "Known facts (use these to answer):" to "Context:"), the sanitizer silently stops catching those leaks. A more robust approach would derive patterns from the actual prompt template, or at minimum co-locate them with the prompt definition.

The system-prompt similarity check uses a separate hardcoded string:

```cpp
static const std::string sysPrompt = "You are MK, a personal AI assistant...";
```

This is a paraphrase, not the actual `MK_SYSTEM_PROMPT`. The two will drift independently.

## HTML escaping injection vector

The `shouldEscapeHtml` function treats any occurrence of a "safe tag" in the text as a signal that the text is intentionally formatted and should not be escaped. But LLM responses routinely contain angle brackets — code snippets (`vector<int>`), comparisons (`a < b`), and even literal HTML tag names when discussing web topics. If the LLM says "use `<code>` for inline formatting", that substring matches the safeTags list and the entire response passes through raw.

The consequence: Telegram's HTML parser receives unescaped `<` and `>` characters from other parts of the same message, causing either a parse failure (Telegram rejects the message) or partial rendering of unintended tags.

The correct pattern is: escape everything first, then apply MK's own formatting to specific spans afterward. The current design conflates "text that MK formatted" with "text that happens to contain tag-like substrings." `sendMessageWithKeyboard` doesn't call `escapeForUser` at all, but its messages are all hardcoded HTML from MK itself, so that's fine — the asymmetry is intentional but undocumented.

## Empty-response propagation gaps

The empty-reply guard at line 2131 in `telegram_poll_loop` catches the final path:

```cpp
if (reply.empty()) {
    reply = "I'm processing that but couldn't generate a response. Try again or rephrase.";
}
```

But `sanitizeLLMResponse` can now return empty in `synthesizeResponse` (line 1057) and in `handle_natural_query` (line 1283/1285). In `synthesizeResponse`, if both local and cloud responses are sanitized to empty, the function returns empty to its caller (`generate_ai_response`), which then falls through to the smart router. The smart router path also calls `sanitizeLLMResponse` (line 1917). If all paths sanitize to empty, the final empty-reply guard catches it — but only in the Telegram poll loop. The CLI path (`handle_natural_query` called from the REPL) has no such guard and will print nothing.

This isn't a regression (the CLI path could already produce empty output), but the new sanitizer makes it more likely to trigger.

## Thinking engine echo detection edge case

`validateThinkingOutput` uses 10-char chunks stepped through the input:

```cpp
size_t chunkSize = std::min((size_t)10, lowerInput.size());
for (size_t i = 0; i + chunkSize <= lowerInput.size(); i += chunkSize) {
```

For inputs shorter than 10 characters (but longer than 5, due to the guard), `chunkSize` equals `input.size()`, so only one chunk is checked. A single full match hits 100% similarity, which correctly rejects echoes. But for inputs of exactly 6-9 characters, the echo detection works as a binary "is the entire input present in the output" check rather than a proportional similarity measure. An output like "Sure! hello there, how can I help?" would be rejected for input "hello th" because the 8-char chunk matches — a false positive for conversational responses that naturally contain the user's words.
