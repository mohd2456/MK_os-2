# Living AI upgrade: crypto trading, multi-device mind, homelab orchestration

MK OS grows from a local knowledge-graph REPL into an autonomous multi-subsystem agent. Five new subsystems land in a single pass: a crypto trading pipeline (market data through execution), a "mind" layer for goal-driven autonomous behavior, a homelab controller (SSH + Docker), a multi-device sync protocol, and ~15,000 lines of pipe-delimited knowledge facts. The architecture remains a single compilation unit (`mk_entry.cpp` includes everything) which keeps linking trivial but makes every module globally visible.

**Watch for:** API credentials stored in plaintext config files with no permission checks (confirmed). SSH commands constructed via string concatenation with no input sanitization, creating shell injection surface (confirmed). The device-comm shared-secret auth is never verified on incoming requests—`authenticated` field exists but nothing sets it based on a challenge (confirmed). Crypto exchange API has a custom HMAC-SHA256 implementation that hasn't been validated against test vectors (likely).

## High-level view

The crypto subsystem chains eight modules from price fetching through risk management to order execution, defaulting to paper-trade mode so nothing fires live without explicit opt-in. The risk manager enforces position caps, drawdown pauses, and stop-losses, but the exchange API's hand-rolled SHA-256 implementation is untested against known-answer vectors, and API secrets sit in a plaintext config file loadable by any process on the machine.

The mind layer (mastery network, goal engine, strategy planner, self-funding, autonomous learner, knowledge validator) creates a Think-Act-Evaluate-Learn loop scheduled via daemon jobs. No subsystem actually executes actions beyond in-memory bookkeeping—the autonomous learner, for instance, doesn't fetch external content yet—so the architecture is a skeleton with safe no-op behavior.

The homelab controller wraps SSH and Docker operations behind classes that pipe commands through `popen`. Input to remote commands is not escaped, so any fact or device name containing shell metacharacters can inject arbitrary commands on remote hosts. The resource monitor properly refuses tasks when CPU, RAM, or temperature exceed thresholds, implementing the "don't crash" design goal.

Multi-device sync uses eventual consistency with a "user corrections always win" conflict rule. Communication goes over plain HTTP with a shared secret, but the auth check is never applied to incoming messages—an endpoint struct carries a `shared_secret` field that nothing validates on receipt.

Knowledge expansion adds 13 bulk `.mk` files totaling ~7,000+ new pipe-delimited facts across domains from physics to DevOps. These are static data files with no executable risk, and the pattern-graph loader handles them via the existing directory scan.

## Crypto pipeline: paper-safe but secrets exposed

The trading bot defaults to `MKBotMode::PAPER` and the exchange API refuses live orders unless `isInitialized()` returns true (requires non-empty API key). This is a reasonable safety gate. The risk manager enforces multiple layers—position size cap at 20%, daily loss limit, drawdown pause at 10%, volatility-adjusted sizing—and the trading bot checks `isTradingPaused()` before every scan cycle. Stop-losses trigger sell orders before new signals are evaluated.

The HMAC-SHA256 implementation in `exchange_api.cpp` (lines 80–180) is a from-scratch SHA-256 plus HMAC wrapper. It avoids an OpenSSL dependency but introduces correctness risk: there are no unit tests comparing output against NIST's published test vectors. A single off-by-one in the padding or round logic would produce valid-looking but wrong signatures, causing silent order failures or—worse—replayed requests if the exchange rejects and the bot retries with stale nonces.

API keys load via `exchangeApi_.initialize("crypto_config.txt")` which reads the file without checking permissions. On a multi-user Linux host (or if the repo is cloned with world-readable defaults), any process can harvest the keys.

## SSH and Docker: shell injection surface

`MKSSHController::buildSSHCommand` concatenates user-supplied `remote_cmd` directly into a shell string:

```cpp
cmd << " '" << remote_cmd << "'";
```

If `remote_cmd` contains a single quote (e.g., from a goal description, device name, or knowledge fact piped into a remote task), the quoting breaks and allows arbitrary command execution on the target host. The Docker manager suffers the same pattern via its `executeOnDevice` calls that pass container names and compose paths through the SSH controller. This is the highest-severity concern for any deployment where MK delegates tasks to real machines.

## Device-comm authentication gap

`MKCommEndpoint` stores `shared_secret` and `authenticated` fields. The message struct includes `auth_token`. But the class never contains logic to:
1. Generate a token from the shared secret and request body.
2. Validate an incoming token against the stored secret.

Any device on the LAN that discovers the REST port can push arbitrary facts via `/api/sync` or delegate tasks via `/api/task`. Combined with the SSH injection surface, an attacker on the local network could achieve remote code execution on all registered homelab nodes.

## CRYPTO/HOMELAB/MIND route dispatch is a pass-through

The smart router correctly identifies crypto/homelab/mind queries via keyword scoring, but in `mk_entry.cpp`'s `handle_natural_query`, all three new route types execute identical code:

```cpp
case MKRouteType::CRYPTO: {
    auto results = sys.graph.getAll(input);
    ...
}
```

None of them invoke `cryptoTradingBot->runScanCycle()`, `marketData_.fetchPrice()`, `deviceRegistry`, `goalEngine`, or any subsystem-specific logic. The REPL user gets the same graph-lookup behavior regardless of routing. The subsystems are only reachable via scheduled daemon jobs (every 5–15 minutes) or via Telegram commands in the polling thread.

## Daemon scheduling and resource gating

New daemon jobs are registered with reasonable intervals (crypto scan every 5 min, device health every 1 min, autonomous learning every 30 min, goal evaluation every 15 min, portfolio rebalance every hour). The autonomous learning job checks `config.isLearningHour()` before running, and the resource monitor's `canRunTask()` is available for pre-flight checks. The "don't crash" philosophy is architecturally present but only enforced in the resource monitor; the daemon itself has no CPU/memory guard around job execution.

## Knowledge sync conflict model

Conflict resolution follows a clear hierarchy: user corrections always win (checked via `user_correction` flag), otherwise latest timestamp wins. Deduplication uses a generated `fact_id` from source+relation+target hash. The sync queue has a configurable max size. The design is sound for eventual consistency, but without the auth layer working, any peer can inject "user corrections" that override legitimate facts.

## Test coverage

26 new integration tests cover: RSI/MACD/Bollinger calculations with boundary cases, risk manager position rejection, portfolio tracking, signal generation logic, router keyword routing for new routes, device registry CRUD, resource monitor threshold logic, knowledge sync conflict resolution, autonomous learner session lifecycle, and config loading. All 37 tests pass with 305 assertions.

Not tested: SSH command construction (no injection test), HMAC-SHA256 correctness against known vectors, trading bot behavior when market data returns invalid results mid-analysis, device-comm message authentication, Docker manager error paths, the Telegram command handlers for `/crypto`, `/devices`, `/goals`.
