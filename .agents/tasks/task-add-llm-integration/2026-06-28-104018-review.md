# Local LLM integration with hardware-gated model management

MK OS gains the ability to detect a local LLM server (llama.cpp on port 8080 or Ollama on port 11434), report hardware sufficiency at boot, and use the server for text cleanup when available. The integration adds a `model_manager.cpp` that reads `/proc/meminfo` to determine free RAM, an `llm_engine.cpp` that probes both servers and provides a `generate()` interface, shell scripts for model download and server launch, and a rewrite of the input preprocessor's `preprocessWithModel()` to actually call both backends instead of stubbing out. If RAM is below 2 GB or no server responds, the system falls back silently to rule-based processing.

**Watch for:** Prompt injection via unsanitized user input concatenated into LLM prompts (confirmed). The JSON escape routine handles syntax validity but the prompt itself has no system/user boundary, so crafted input can override the "clean up" instruction. The macOS RAM path returns total RAM * 0.6 rather than actual available memory (confirmed). The preprocessor contains ~150 lines of near-identical code for the two server paths with no shared abstraction (confirmed).

## High-level view

The hardware gate reads `MemAvailable` from `/proc/meminfo`, compares against 2 GB, and disables LLM features if insufficient. The check runs once at construction time and isn't re-evaluated, so a system that frees memory after boot won't benefit until `refresh()` is explicitly called.

The dual-server detection probes llama.cpp first, Ollama second, and stops at the first responding endpoint. The engine stores the result at construction and exposes `refresh()` for re-detection, though nothing in the boot sequence calls it after initial construction.

The preprocessor rewrite replaces a stub that always fell back to rule-based processing with a full implementation that sends user input to whichever server responds, parses the JSON response with a hand-rolled escape-aware parser, and feeds the cleaned text back through intent detection. The two server paths are structurally identical except for the URL, JSON field names, and one extra parameter.

The tests verify the offline path: RAM detection returns positive values, `isAvailable()` is false when no server runs, and `preprocessWithModel()` falls back cleanly. No tests exercise the happy path because no server runs in the test environment.

<details>
<summary>Issues (5)</summary>

1. **Prompt injection in preprocessor** — User input is concatenated directly into the LLM prompt string with no system/user delimiter or escaping beyond JSON special characters. A user can append instructions that override the "clean up" directive. Add a system prompt or use a chat template that separates roles.
2. **Duplicated server interaction logic** — The two 75-line blocks in `preprocessWithModel()` for llama.cpp and Ollama are structurally identical. Extract a shared helper that takes the URL, field name, and optional extra parameters.
3. **macOS RAM detection is a guess** — `readAvailableRAM()` on macOS returns `hw.memsize * 0.6`, which measures total physical memory, not available memory. Use `vm_stat` or `host_statistics64` for actual free pages.
4. **Ollama model name hardcoded** — The engine and preprocessor both hardcode `"tinyllama"` as the Ollama model name. If the user runs a different model, generation requests will fail. Consider reading the model list from `/api/tags` or making the name configurable.
5. **No timeout on detection during boot** — `MKLLMEngine` constructor calls `detectServer()` which performs two HTTP requests with 1-second timeouts each. On a network where these ports are filtered (not refused), boot adds up to 2 seconds of blocking delay. Acceptable for now, but worth noting for future async init.

## Prompt construction and injection surface

The preprocessor builds LLM prompts by concatenating a fixed instruction string with raw user input:

```cpp
std::string prompt = "Clean up and fix this text. Only return the cleaned text, nothing else: " + input;
```

The input is JSON-escaped (quotes, backslashes, newlines, tabs) before embedding in the POST body, but this only handles JSON syntax validity. There's no system/user role separation: llama.cpp's `/completion` endpoint accepts a flat prompt, and the Ollama call uses the raw `prompt` field rather than a chat-format `messages` array. The model has no structural way to distinguish the fixed instruction from user-supplied content.

The security impact is bounded because the LLM output is only used for text cleanup and then fed through rule-based intent detection — the worst case is garbage text that gets low confidence from `detectIntent()`. Worth fixing if the LLM path is ever used beyond cosmetic cleanup. (confirmed)

## Code duplication in preprocessWithModel()

The llama.cpp path and the Ollama path share identical structure: init curl, build JSON with escaped prompt, set headers, perform request, parse response string with escape-aware loop, construct `MKPreprocessResult`. The only differences:

- URL: `/completion` vs `/api/generate`
- Request JSON: `n_predict` + `temperature` vs `model` + `stream`
- Response field: `"content"` vs `"response"`

The `MKLLMEngine` class already has a `generate()` method that does exactly this with its own JSON response parsing. The preprocessor doesn't use it, presumably because it wants the 10-second timeout and 128 token limit rather than the engine's 30-second / 256 token defaults. A refactor that lets the preprocessor call the engine (or a shared lower-level function with configurable parameters) would cut ~150 lines to ~10. (confirmed)

## Hardware detection correctness

The macOS path calls `sysctl -n hw.memsize` (total physical RAM, not available) and multiplies by 0.6. On a Mac running heavy workloads, actual available memory could be well below 60%, causing the system to falsely report hardware as sufficient. Since the project runs on Linux (confirmed by test output), this is a portability correctness issue rather than a current regression. (confirmed)

The `MKModelManager` constructor prints `[MODEL MANAGER] Hardware check complete.` unconditionally, and the `MKLLMEngine` constructor prints `[LLM ENGINE] Initializing...`. Since `MKSystem` constructs `MKLLMEngine` as a member, these messages fire during struct construction — before `checkAndReport()` is called in `main()` and potentially before the boot banner. This produces unsequenced output. (likely)

## Shell scripts

`start_server.sh` searches for `llama-server` in PATH, then in the script directory, then tries the bare name `server` as a last resort. The `server` binary name is overly generic and could match unrelated programs on some systems. (possible)

## Test coverage gaps

Not tested: the JSON response parsing logic (hand-rolled escape handling with edge cases around malformed JSON, truncated responses, nested objects), the prompt construction path, behavior when a server *is* available, the Ollama model name mismatch scenario, or the macOS RAM path.
