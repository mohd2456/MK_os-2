# Thread safety, JSON parser, and RAII fixes for MK OS unified entry (v2)

The fix commit addresses the four blocking issues raised in v1: a `std::mutex` now guards shared state between the Telegram polling thread and the REPL, the JSON update parser bounds field searches to the current update object, the raw `MKTelegram*` is replaced with `std::unique_ptr<MKTelegram>`, and natural language Telegram messages now route through `generate_ai_response()` instead of echoing. The project builds cleanly and the test suite passes (7 tests, 62 assertions, 0 failures).

**Watch for:** The mutex is held during the blocking `getUpdates` HTTP call (5-second timeout), which stalls the REPL if a user types a command during a poll cycle (confirmed). The REPL's GRAPH-route vector-search fallback still concatenates raw `sourceText` strings without routing through the composer, an asymmetry with `generate_ai_response` which does compose properly (confirmed). TLS certificate verification remains disabled on all Telegram API calls (confirmed, pre-existing but now higher risk since the Telegram channel surfaces knowledge-graph contents).

**Verdict**: APPROVED

## High-level view

The synchronization strategy is a single coarse-grained `std::mutex` on `MKSystem`. The REPL acquires it for every command and natural-language query; the Telegram thread acquires it for `getUpdates` and again for message processing. This eliminates data races but introduces a latency coupling: the REPL blocks while the HTTP poll is in flight (up to 5 seconds per cycle). For a single-user local system this is tolerable; it would need restructuring if the Telegram polling timeout were increased or if the REPL were expected to remain responsive during network delays.

The JSON parser fix correctly bounds field searches to the current update object via `updateBoundary`. The parser still does not handle escaped quotes inside message text robustly (a `\"` inside the text body terminates extraction early), producing truncated but not misrouted messages.

The REPL's `handle_natural_query` still has one path (GRAPH route, vector-search fallback at line 644) that concatenates raw edge strings directly into the response, while the equivalent path in `generate_ai_response` routes through the composer. The same query produces different quality responses depending on whether it arrives via Telegram or the local terminal.

## Mutex held across blocking I/O

The polling loop acquires `systemMutex` to call `getUpdates`:

```cpp
std::string response;
{
    std::lock_guard<std::mutex> lock(sys.systemMutex);
    response = sys.telegram->getUpdates(lastUpdateId);
}
```

`getUpdates` constructs a URL with `timeout=5` (Telegram long-polling), meaning the HTTP call can block for up to 5 seconds waiting for new messages. During that window, any REPL command that tries to acquire `systemMutex` (which is all of them except `/help` and `/quit`) will block the user's terminal.

The fix is straightforward: call `getUpdates` without holding the lock (it only reads `sys.telegram` which is never reassigned after construction), then acquire the lock only for the message-processing section that calls `generate_ai_response` and `sendMessage`. The `telegram` pointer stability is guaranteed because `MKSystem` is never moved or copied after construction.

## REPL vector-search fallback produces raw edge dumps

In `handle_natural_query`, the GRAPH route's innermost fallback:

```cpp
auto vecResults = sys.vectorSearch.search(input, 3);
if (!vecResults.empty() && vecResults[0].score > 0.3f) {
    response = "";
    for (const auto& vr : vecResults) {
        if (vr.score > 0.3f) {
            response += vr.sourceText + " ";
        }
    }
    confidence = vecResults[0].score;
    answered = !response.empty();
}
```

This produces output like `"python created_by guido_van_rossum python is_a programming_language"` — raw graph-edge notation returned to the user. The Telegram path's equivalent code in `generate_ai_response` constructs an `MKResponseContext` and calls `sys.composer.composeAnswer(ctx)`, producing a human sentence. The REPL path predates the fix commit and wasn't updated to match.

## TLS verification disabled

`telegram.cpp` line 33:

```cpp
curl_easy_setopt(curl, CURLOPT_SSL_VERIFYPEER, 0L);
```

Pre-existing, but the risk profile changed: before the fix, Telegram only echoed messages. Now it routes queries through the AI pipeline, meaning an attacker performing a MITM on the Telegram API connection could inject arbitrary queries and receive knowledge-graph responses, effectively exfiltrating the knowledge base. The comment says "2010 compatibility helper" but the system targets modern macOS/Linux where certificate chains work fine.
