# Unifying MK OS into a single-entry boot with full module wiring

MK OS was a collection of disconnected modules each with their own `main()`. This change consolidates everything into a single compilation unit (`mk_entry.cpp`), wires all mk_brain subsystems (learning, memory, embeddings, vector search, cache, briefing, fact extraction, reasoning), adds Telegram polling in a background thread, expands the knowledge base by 550+ facts across three new `.mk` files, fixes the Makefile to use single-compilation-unit linking, and adds a test suite covering the core AI pipeline. The hardcoded Telegram token was removed and replaced with `MK_TELEGRAM_TOKEN` env var loading.

**Watch for:** Thread-safety gap between the Telegram polling thread and the REPL both mutating `MKSystem` without synchronization (confirmed). The Telegram JSON parser is brittle and will mismatch fields across update boundaries in multi-update responses (confirmed). The `MKTelegram` object leaks if construction succeeds but the env var token is empty-string at API call time (possible).

## High-level view

Dead `main()` entry points are deleted or gutted, the Makefile switches from a broken multi-object link to compiling only `mk_entry.cpp`, and the module wiring brings all subsystems under one struct's lifecycle. The token credential that was hardcoded in two files is gone; all Telegram auth now routes through `std::getenv("MK_TELEGRAM_TOKEN")`.

The Telegram polling thread shares the entire `MKSystem` reference with the main REPL thread. Neither the graph, learning engine, memory, nor fact extractor provides any synchronization. Under concurrent load (user typing in the REPL while Telegram messages arrive), data corruption is possible on any shared container.

The JSON parsing in the polling loop searches for field markers (`"update_id":`, `"chat"`, `"text":"`) with positional offsets but does not scope each search to the current update object. When the API returns multiple updates in one response, a `"text"` field from update N can be associated with the `chat_id` from update N+1.

The test suite covers the correct modules in isolation but does not test any of the newly-wired integration logic in `mk_entry.cpp` (Telegram polling, service lifecycle, vector-search fallback path, dual-write learning).

<details>
<summary>Issues (5)</summary>

1. **No synchronization on shared MKSystem** — The Telegram thread and REPL thread both read/write `sys.telegram`, `sys.factExtractor`, `sys.memory`, etc. concurrently. Add a mutex around shared state or isolate the Telegram thread's data, posting results to the main thread via a queue.
2. **JSON parser cross-contaminates multi-update responses** — The `"text"` and `"chat"` field searches are not bounded to the current update object. Scope each field search between the current `update_id` position and the next `update_id` position (or end of string).
3. **`MKTelegram` raw pointer without RAII** — `sys.telegram` is `new`'d in the constructor and `delete`'d in the destructor, but if anything throws between construction and destruction (or if MKSystem is copied), the pointer leaks or double-frees. Use `std::unique_ptr<MKTelegram>`.
4. **Telegram poll only echoes messages** — Natural language messages get a static "MK received: " echo instead of being routed through `handle_natural_query`. The wiring comment and feature spec both suggest full AI pipeline routing, but only `/` commands get meaningful responses.
5. **Integration paths not tested** — The test suite covers isolated modules but not the vector-search fallback in the GRAPH route, the learning-engine dual-write from `/learn`, the biographical extraction from natural queries, or the service manager lifecycle.

## Thread safety between Telegram polling and REPL

The `telegram_poll_loop` function takes `MKSystem& sys` and runs on a separate thread. It calls `sys.telegram->getUpdates()`, `sys.telegram->sendMessage()`, and `sys.telegram->autoReply()`. Meanwhile the main REPL thread calls `handle_natural_query(sys, ...)` which mutates `sys.graph`, `sys.memory`, `sys.factExtractor`, `sys.learningEngine`, `sys.brainMemory`, and `sys.vectorSearch`.

None of these module classes use internal mutexes. The `std::vector<MKEdge> edges` inside `MKPatternGraph` is read by the REPL's graph queries and could be modified by `persistNewFact()` concurrently. Today the polling thread only echoes or calls `autoReply`, but the shared reference to the full system is the hazard. If Telegram polling is extended to route messages through the AI pipeline (which the feature spec states is the intent), concurrent mutation is inevitable without a lock or message-passing queue.

## Telegram JSON parsing breaks on batched updates

The polling loop parses the Telegram Bot API response by searching for field markers sequentially:

```cpp
size_t uidPos = response.find("\"update_id\":", searchPos);
// ... extract updateId ...
size_t chatPos = response.find("\"chat\"", numEnd);
// ... extract chatId ...
size_t textPos = response.find("\"text\":\"", numEnd);
```

The `chatPos` and `textPos` searches start from `numEnd` (end of the update_id number) but are not upper-bounded by the next update_id. When the API returns two updates like `[{update_id:1, message:{chat:{id:A}, text:"hello"}}, {update_id:2, message:{chat:{id:B}, text:"world"}}]`, the search for `"text"` starting from update 1's position could skip past update 1's text (if it's absent because the message is a photo or sticker) and find update 2's `"text":"world"`, associating it with chat A. The result: messages get sent to the wrong chat.

Fix: find the position of the *next* `"update_id":` and use it as the upper bound for all field searches within the current update.

## MKTelegram lifecycle and the raw pointer

```cpp
MKTelegram* telegram;
// in constructor:
telegram = new MKTelegram();
// in destructor:
delete telegram; telegram = nullptr;
```

`MKSystem` has no copy/move constructors, so an inadvertent copy would double-free. If any initializer *after* the `new MKTelegram()` line throws, the destructor won't run and the pointer leaks. Switching to `std::unique_ptr<MKTelegram>` eliminates both issues with zero runtime cost.

## Vector-search fallback produces raw edge text

When the GRAPH route finds no direct facts and no path-query results, the code falls through to:

```cpp
auto vecResults = sys.vectorSearch.search(input, 3);
if (!vecResults.empty() && vecResults[0].score > 0.3f) {
    response = "";
    for (const auto& vr : vecResults) {
        if (vr.score > 0.3f) response += vr.sourceText + " ";
    }
}
```

The `sourceText` field contains the concatenated `source + " " + relation + " " + target` string from indexing time (e.g. "python created_by guido_van_rossum"). Multiple results are concatenated with spaces and returned directly to the user, producing incoherent dumps like "python created_by guido_van_rossum python is_a programming_language python has_feature dynamic_typing". The composer is available and could wrap these facts into a sentence, but it isn't used in this path.

## Test coverage gaps

**Not tested:** The vector-search fallback path, the dual-write in `/learn`, the biographical extraction call in `handle_natural_query`, the service manager `start_all()`/`shutdown_all()` lifecycle, the Telegram polling loop, the daemon job registration, and the graceful shutdown sequence. These are the integration paths most likely to regress silently.

## Knowledge base weight uniformity

All weights in the three new `.mk` files are `1.0`. The existing `core_facts.mk` and `world_knowledge.mk` use varying weights (0.85-1.0) to express confidence. A uniform weight of 1.0 removes the graph's ability to rank facts by reliability during path queries and vector-search scoring — worth noting if confidence-weighted ranking matters downstream.
