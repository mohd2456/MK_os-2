# Three-layer thinking/decision architecture with multi-provider routing

MK's conversation pipeline gains a three-layer architecture: a thinking engine (Layer 1) sends compact reasoning prompts to a small model via the provider router, a decision engine (Layer 2) parses the thinking output for action keywords and dispatches tools, and a context layer (Layer 3) gathers graph facts, conversation history, and system state. The provider router manages five backends (Groq, OpenRouter, Nvidia NIM, HuggingFace, Ollama) with urgency-based scoring, daily quota tracking, and health monitoring. API keys are stored with XOR obfuscation tied to hostname+username. The graceful fallback chain goes thinking -> direct LLM -> Genesis -> graph facts + templates.

**Watch for:** The `/setkey` Telegram command transmits API keys in plaintext over the Telegram Bot API (confirmed). The decision engine's keyword-based action parsing will trigger false positives on common words like "tell", "stop", "memory", "system" appearing in any thinking output (confirmed). The XOR key encryption provides no real security since the seed is deterministically derived from hostname and username, both trivially discoverable (confirmed). The `shouldConfirm` check in the decision engine runs on the thinking output, not the user input, meaning a model that says "no need to restart" still triggers the destructive confirmation gate (likely).

## High-level view

The three-layer design separates reasoning from execution: the thinking engine asks a 3B model "what should MK do?" with max_tokens=100, then the decision engine pattern-matches that reasoning for tool keywords and dispatches accordingly. When thinking returns empty, the system degrades through direct LLM, Genesis, graph facts with templates, and finally bare templates. Both the CLI (`handle_natural_query`) and Telegram (`generate_ai_response`) paths implement the same architecture with full parity including evolution engine feeds.

The decision engine's keyword-to-action mapping is the weakest abstraction in the stack. Common English words in the registry ("tell", "explain", "system", "stop", "resource") will fire on thinking outputs that merely mention these concepts without intending the corresponding tool action. A thinking output like "User wants to know about system design patterns" would dispatch `SYSTEM_INFO` and return CPU/RAM stats instead of answering the question.

The real security exposure is the `/setkey` command: API keys transit Telegram's HTTP API in cleartext messages, are stored in Telegram's chat history on their servers, and anyone with access to the chat can scroll up and read them. The XOR encryption on disk adds no meaningful protection since the seed derives from public system identifiers.

<summary>Issues (7)</summary>

1. **API keys transmitted via Telegram chat** — `/setkey groq gsk_abc123` sends the key as a plaintext Telegram message. The key is stored in Telegram's chat history, visible to anyone with access to the chat, and transits Telegram's servers. Consider accepting keys only through a local CLI command or environment variable, or at minimum delete the user's message immediately after processing.
2. **Decision engine false-positives on common words** — The tool registry maps "tell", "explain", "memory", "system", "resource", "stop" to tool actions. Any thinking output containing these words triggers tool dispatch. "User should stop worrying" would trigger `CONFIRM_DESTRUCTIVE`. Require more specific multi-word patterns or require tool keywords to appear as intent markers, not incidental words.
3. **`shouldConfirm` runs on thinking output, not user input** — If the model's reasoning says "no need to restart anything", the word "restart" still triggers the destructive confirmation gate. The check should either run on the user's original input or require the thinking output to express intent (e.g., "should restart" vs. mentioning "restart" in any context).
4. **XOR "encryption" seed is trivially reproducible** — The seed derives from `gethostname()` + `$USER` via DJB2. Any process on the same machine (or anyone who knows the hostname and username) can reproduce the seed and decrypt all stored keys. This is fine for casual obfuscation but should not be described as "encrypted" in user-facing messages like "key verified and saved" without caveat.
5. **Request logger logs thinking provider attribution incorrectly** — In `generate_ai_response`, the thinking request is logged with `sys.providerRouter.getActiveProvider()` which returns the first online provider in array order, not the actual provider the thinking engine selected via `pickThinkingProvider()`. The logged provider may not match the one that actually served the request.
6. **Thinking engine Ollama JSON body lacks prompt escaping** — The Ollama-format JSON body embeds the prompt with no escaping (`"\"content\":\"" + prompt + "\""`). The prompt is built from user input and graph facts. If any contain quotes or backslashes (which the OpenAI path handles), the JSON will be malformed and the request fails silently.
7. **Token estimate heuristic (`input.size() / 4`) used for quota tracking** — The request logger records "tokens_used" as character count divided by 4. For non-English input or code snippets, this diverges significantly from actual token counts, potentially causing quota miscalculations.

## Decision engine keyword collision surface

The tool registry in `decision_engine.cpp` maps 28 keywords to actions. Several are single common English words that will appear routinely in LLM reasoning output:

```
"tell" -> EXPLAIN
"stop" -> CONFIRM_DESTRUCTIVE  
"system" -> SYSTEM_INFO
"resource" -> SYSTEM_INFO
"explain" -> EXPLAIN
"describe" -> EXPLAIN
"fact" -> GRAPH_LOOKUP
"knowledge" -> GRAPH_LOOKUP
```

The `parseThinkingOutput` method converts the entire thinking output to lowercase and does substring matching. For destructive keywords the consequence is significant: "User doesn't want to stop watching, they want more episodes" triggers `shouldConfirm` on "stop" and the user gets a "destructive action" confirmation prompt for a benign request.

The `shouldConfirm` method has its own separate keyword list that overlaps with but differs from the tool registry. It runs first in `process()`, before `parseThinkingOutput`, so even if the tool parsing would have classified the action as CONVERSATIONAL, the confirmation gate fires independently.

## Thinking engine Ollama JSON injection

The Ollama request body construction does not escape the prompt:

```cpp
body = "{\"model\":\"phi3:mini\","
       "\"messages\":[{\"role\":\"system\",\"content\":\"You are a reasoning engine. "
       "Respond in 1-2 sentences only. State what action should be taken.\"},"
       "{\"role\":\"user\",\"content\":\"" + prompt + "\"}],"
       "\"stream\":false,...}";
```

The `prompt` variable is built from `buildThinkingPrompt` which concatenates user input, graph facts, and system state without escaping. A user input containing a literal `"` character (e.g., `what does "hello world" mean`) or a graph fact containing a quote would break the JSON. The OpenAI-compatible path (used for Groq, OpenRouter, etc.) has explicit escape handling for `"`, `\n`, `\r`, `\t`, and `\\`, but the Ollama path skips this entirely.

## Provider attribution in request logging

In the Telegram `generate_ai_response` function:

```cpp
std::string thinkProvider = sys.providerRouter.getActiveProvider();
sys.requestLogger.logRequest(thinkProvider, "thinking: " + input.substr(0, 40),
                             thinkTokens, thinkLatency, !thinking.empty());
```

`getActiveProvider()` returns the first ONLINE provider in array order (groq first). But `thinkingEngine.think()` internally calls `pickThinkingProvider()` which prefers: ollama > groq > openrouter > any. If Ollama is running and Groq has a key, `getActiveProvider()` returns "groq" while the thinking engine actually used "ollama". Every logged thinking request would be mis-attributed.

## API keys in Telegram chat history

The `/setkey` command flow:
1. User sends `/setkey groq gsk_abc123456789` as a regular Telegram message
2. The message is stored permanently in Telegram's servers and chat history
3. The bot reads the key and stores it via XOR encryption locally
4. The bot replies confirming the key was saved

The key persists in Telegram's message store (accessible to Telegram, any compromised device with the chat open, and any bot admin). There is no message deletion after processing. The Telegram Bot API supports `deleteMessage` — calling it on the user's `/setkey` message after extraction would remove the key from chat history on most clients.
